In one sentence: An API gateway is a single entry point that sits between external clients and your services — it routes each request to the right service, can aggregate several calls into one, translates protocols, and handles cross-cutting concerns like authentication, rate limiting and TLS.

Use when

  • External clients (mobile, web, partners) need to talk to many fine-grained services.
  • You want to hide internal topology and decouple clients from service decomposition.
  • Cross-cutting concerns (auth, throttling, TLS) should be handled in one place.

Avoid / be careful when

  • You have a single client and a couple of services — the gateway is overhead.
  • You're tempted to put business logic in it (that's the road to a new monolith).
  • You can't make it highly available — it becomes a single point of failure.

The problem: clients shouldn't talk to 30 services directly

Once a system is decomposed, a single screen often needs data from many services. The FTGO mobile app's "order details" view needs the order, the restaurant, the delivery status and the bill — spread across four services. If the client calls each service directly, you get real problems:

  • Chatty, high-latency clients. A mobile device on a flaky network making many round-trips is slow and battery-hungry.
  • Tight coupling. Clients now know your internal service URLs and decomposition; refactoring a service breaks every client.
  • Protocol mismatch. Internal services may speak gRPC or AMQP, but a browser wants JSON over HTTP.
  • Duplicated concerns. Auth, rate limiting and TLS would have to be re-implemented per client/service.

The pattern: a single, smart front door

Introduce one service — the API gateway — that is the single entry point for all external requests. It is responsible for request routing, API composition (calling several services and combining results), and protocol translation, and it offloads cross-cutting concerns.

How it works

A gateway has four core jobs. The first three are request-handling; the fourth is "edge functions" it performs on every request.

  1. Routing. Map an external path (GET /orders/123) to an internal service. A pure reverse-proxy gateway does only this.
  2. API composition. For coarse-grained client needs, the gateway invokes several services and combines the results — an application of the API composition pattern.
  3. Protocol translation. Expose REST/GraphQL to the outside while talking gRPC/messaging inside.
  4. Edge offloading. Authentication, authorisation, rate limiting, caching, TLS termination, request/response logging and metrics — done once, for everyone.
💡
Gateway vs. Backends for Frontends

One generic gateway serving every client tends to bloat. The Backends for Frontends (BFF) variation gives each client type its own gateway (mobile BFF, web BFF, partner BFF), each owned by that client's team. Same idea, better ownership and tailoring. Start with one gateway; split into BFFs when teams collide.

Worked use cases

Use case · FTGO mobile

One round-trip instead of four

The mobile "order status" screen needs four services. The gateway exposes a single coarse-grained GET /orders/{id} that fans out, joins, and returns one payload — turning four mobile round-trips into one, with partial-failure handling (return delivery: null if Delivery times out rather than failing the screen).

Use case · E-commerce

Product page aggregation + per-consumer rate limits

A product page composes catalog, pricing, inventory and ratings. The gateway aggregates them and applies a tighter rate limit on the /payment route than on /catalog, validating the JWT once and forwarding a trusted X-User-Id header downstream.

Use case · Banking / open APIs

Partner platform with OAuth2 and audit

A bank fronts all partner-facing APIs with a managed gateway (e.g. Apigee). Partners register apps, get API keys, are throttled per plan, authenticate via OAuth2 client-credentials, and every call is logged for compliance — none of which touches the internal services.

Trade-offs

Benefits

  • Encapsulates internal structure; clients are decoupled from decomposition.
  • Reduces round-trips and tailors APIs per client.
  • One place for auth, throttling, TLS, observability.
  • Protocol translation lets you use the best internal protocol.

Drawbacks

  • One more highly-available component to build, deploy and operate.
  • Can become a development bottleneck if every team must edit it.
  • Risk of becoming a "god service" full of business logic.
  • Adds a network hop (mitigated by parallel fan-out + caching).

In the wild — tooling & who uses it

  • Kong — open-source/enterprise, plugin-based (auth, rate-limit, logging); used by large enterprises.
  • Spring Cloud Gateway — reactive JVM gateway; predicate + filter routes; integrates with Spring Security.
  • AWS API Gateway / Azure API Management / Google Apigee — fully managed gateways with throttling, developer portals and (for Apigee) API monetisation used by major banks.
  • Envoy — the L7 proxy under many modern gateways and ingress controllers (Gloo, Contour, Emissary).
  • Netflix Zuul — the historically important gateway; Zuul 2 went async on Netty. Largely superseded by Envoy-based stacks in new builds.
  • GraphQL (Apollo Federation, Netflix DGS) — a modern way to implement a composing gateway/BFF where each client asks for exactly the fields it needs.

Pitfalls & anti-patterns

⚠
Watch out for these
  • The "god gateway". Accumulating routing + transformation + orchestration + business rules rebuilds a monolith at the edge.
  • Single point of failure. One un-clustered instance with no health checks/auto-scaling is your system's weakest link.
  • Auth done twice or not enough. Validating that a token merely exists without checking issuer, audience, expiry and scopes is a common security gap.
  • Deployment coupling. If adding any endpoint needs a manual gateway change, the gateway becomes an org-wide bottleneck — prefer declarative, version-controlled routes.

Key takeaways

  • The gateway is the system's front door: route, compose, translate, and offload edge concerns.
  • Keep it thin — coordination and cross-cutting concerns only, never domain logic.
  • Make it highly available; it's on the critical path of every request.
  • When one gateway serves clashing clients, split into BFFs.
  • It naturally hosts token validation, circuit breakers and API composition.
Check yourself

A teammate proposes putting the "calculate loyalty discount" rule into the API gateway because "all clients need it". Why push back?

Show answer

That's business logic. It belongs in a service (owned by a team, testable, reusable by all entry points including async flows). Putting it in the gateway couples a domain rule to the edge, makes the gateway a shared bottleneck every team must edit, and means non-gateway callers silently miss the rule. The gateway should route to the service that owns the rule, not own it.
