Asynchronous Messaging

Decouple services with a message broker; improve availability with event-driven communication.

Messaging
In one sentence: Services communicate by sending messages through a message broker instead of calling each other directly — the sender doesn't wait for or even know the receiver, which decouples them in time and dramatically improves availability.

Use when

  • You want temporal decoupling — sender and receiver needn't be up together.
  • You're publishing events many services react to.
  • Work is long-running, bursty, or fire-and-forget.

Be careful when

  • The caller truly needs an immediate answer (use RPI).
  • The team isn't ready for eventual consistency & async debugging.
  • You forget idempotency — messages arrive at least once.

The problem: synchronous coupling is fragile

As the RPI lesson showed, synchronous request/response couples the availability of caller and callee and adds latency to every chain. For everything that doesn't need an immediate reply, there's a better way.

The pattern: talk through a broker

The sender writes a message to a channel on a message broker; the broker delivers it to one or more consumers. The sender returns immediately. If a consumer is down, the broker holds the message until it recovers.

How it works

Message & channel types

  • Message kinds: command (do X), document (here's data), event (X happened).
  • Channels: point-to-point (exactly one consumer handles each message — work queues) and publish-subscribe (every subscriber gets a copy — events).
  • Async interaction styles: one-way notifications, async request/response (via a reply channel), and one-to-many publish/subscribe.

Three rules you cannot skip

  • Idempotency. Brokers deliver at least once; design consumers so reprocessing a message is harmless (track processed message ids).
  • Ordering. Use a partition/ordering key (e.g. orderId) so related messages stay in order.
  • Reliable publishing. Don't lose messages on a crash between "DB commit" and "publish" — use the transactional outbox.

Improving availability by removing synchronous calls

The book's flagship example: instead of Order Service synchronously calling Consumer/Kitchen/Accounting at order time, each service subscribes to events and keeps a local replica of the data it needs. Now Order Service can create an order even if the others are momentarily down — availability stops multiplying downward. This is the saga machinery in disguise.

In the wild — brokers & who uses them

  • Apache Kafka — the de-facto log-based streaming backbone (event sourcing, replay, high throughput 500k–1M+ msg/s). As of Kafka 4.0 (2025) ZooKeeper is gone — KRaft makes it a single-binary deploy.
  • RabbitMQ — rich routing (direct/topic/fanout/headers), great for task queues & competing consumers; lowest latency at moderate scale.
  • Apache Pulsar — separates compute (brokers) from storage (BookKeeper); built-in multi-tenancy, geo-replication, tiered storage.
  • NATS / JetStream — tiny, sub-millisecond; CNCF graduated. AWS SQS/SNS, Google Pub/Sub — managed cloud options.
🧠
Choosing a broker

Need ordered, replayable event streams + consumer groups → Kafka. Need flexible routing and classic work queues with no replay → RabbitMQ. Need the lightest possible footprint → NATS. Need multi-region with tight RPOs → Pulsar.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Non-idempotent consumers — duplicate delivery double-charges, double-ships. Track message ids.
  • Lost messages on the dual-write gap — use the outbox.
  • Ignoring ordering — interleaved events corrupt state; key by aggregate id.
  • No dead-letter handling — poison messages block the queue forever.
  • Async everywhere — some flows really do need a synchronous reply; don't be dogmatic.

Key takeaways

  • Messaging decouples services in time → big availability gains over synchronous RPI.
  • Know your message kinds (command/document/event) and channels (point-to-point / pub-sub).
  • Always design for at-least-once delivery: idempotent consumers + ordering keys.
  • Publish reliably with the transactional outbox; it underpins sagas and event sourcing.
Check yourself

A consumer crashes after doing its work but before acknowledging the message. What happens next, and what must the consumer guarantee?

Show answer

The broker, not having received an ack, redelivers the message — so it's processed twice. The consumer must therefore be idempotent: e.g., record each handled message id in the same local transaction as its effect, and skip messages it has already processed. At-least-once delivery makes idempotency non-negotiable.
