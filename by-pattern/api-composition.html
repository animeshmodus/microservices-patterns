API Composition

Answer a cross-service query by invoking several services and joining the results in memory.

API composition
In one sentence: To answer a query whose data lives in several services, an API composer calls each owning service and joins the results in memory — the simplest way to do a "cross-service query" when you can't run a database join across separate databases.

Use when

  • A query needs data from 2–4 services.
  • The in-memory join is small and cheap.
  • You want the simplest option before reaching for CQRS.

Avoid when

  • The join is large (big in-memory datasets) or N+1.
  • You need rich filtering/sorting across services → use CQRS.
  • Too many dependencies make the query fragile/slow.

The problem: there are no joins across services

In a monolith, "show order 123 with its restaurant, delivery status and bill" is a single SQL join. Once Order, Kitchen, Delivery and Accounting each own a private database, that join is impossible. The data still needs assembling — so something must gather the pieces.

The pattern: a composer fans out and joins

An API composer implements the query by invoking each provider service and combining the responses. The composer can be a dedicated service, or (very commonly) the API gateway itself.

How it works — design choices

  • Call in parallel wherever the calls are independent (reactive/async), so latency is the slowest single call, not the sum.
  • Handle partial failure — return partial data (e.g. delivery: null) rather than failing the whole response; wrap calls in circuit breakers.
  • Keep the composer stateless — never persist intermediate results (that would make it a hidden shared database).
  • Mind availability — the composer depends on all providers, so its availability is the product of theirs. Cache reference data and degrade gracefully.

Trade-offs

Benefits

  • Simplest way to query across services.
  • No extra datastore to maintain (unlike CQRS).
  • Always reads fresh data from the source.

Drawbacks

  • In-memory joins are costly for large datasets.
  • Reduced availability (depends on every provider).
  • Risk of N+1 fan-out; limited filtering/sorting.

Worked use case

Use case · E-commerce product page

Compose catalog + pricing + inventory + ratings

The composer calls four services in parallel and merges them into one ProductCard. Inventory is treated as optional: on a timeout the page shows the product with stockLevel: null rather than failing — partial results beat a blank page.

In the wild

  • The API gateway commonly performs composition ("gateway aggregation"); Microsoft's Azure Architecture Center documents it as a named pattern.
  • GraphQL (Apollo, Netflix DGS) is essentially typed API composition — the client asks for a shape and the server fans out to resolvers.
  • Netflix's API layer historically fanned out to dozens of services to build a single device payload.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Sequential fan-out — calling providers one after another multiplies latency; parallelise.
  • N+1 composition — composing a list by calling a detail service per item; batch instead.
  • Big in-memory joins — pulling huge result sets to join in the composer; that's a sign you need CQRS.
  • Stateful composer — persisting joined results turns it into a shadow database.

Key takeaways

  • API composition answers cross-service queries by invoking providers and joining in memory.
  • Call providers in parallel, handle partial failure, stay stateless.
  • It's the simplest option — but availability and join cost limit it.
  • When it doesn't scale, switch to CQRS read models; the gateway often hosts the composer.
Check yourself

The "order history" screen must filter and sort thousands of a customer's past orders by fields spread across four services. Why is API composition a poor fit here, and what should you use?

Show answer

Filtering/sorting across services would force the composer to pull large datasets from each provider and join/sort them in memory on every request — slow, fragile, and dependent on all four services being up. This is the textbook trigger for CQRS: maintain a denormalised, query-optimised order-history read model kept current by events, so the query hits one indexed store.
