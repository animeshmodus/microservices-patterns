Deployment: Containers, Kubernetes & Serverless

Four ways to run a service: language package, VM, container/Kubernetes, and serverless functions.

Deploy a service as a containerDeploy a service as a VMLanguage-specific packaging formatServerless deployment
In one sentence: There are four ways to package and run a service — as a language-specific package, in its own virtual machine, in a container (orchestrated by Kubernetes), or as serverless functions — trading isolation, startup speed, resource efficiency and operational burden against each other.

Default choice today

  • Containers + Kubernetes for most long-running services.
  • Serverless for event-driven, spiky, or glue workloads.
  • VM for hard isolation / special hardware; package for legacy/simple cases.

Be careful when

  • You treat containers like VMs (SSH in, local state).
  • You underestimate Kubernetes operational complexity.
  • Serverless cold starts sit on a latency-critical path.

The problem: independently deployable means deployable how?

Microservices promise frequent, independent deployment — but only if packaging and runtime make that cheap, fast and safe. The choice shapes startup time, isolation, density, cost and how much platform you must operate.

How it works — the mainstream path

Most teams build a container image (multi-stage Dockerfile, minimal/distroless base), push it to a registry, and let Kubernetes run it: deployments + replicas, service discovery via DNS, autoscaling (HPA), health probes, and zero-downtime rolling updates. Config & secrets come from ConfigMaps/Secrets. A service mesh can separate release from deploy (ship the bits, shift traffic later).

Worked use cases

Use case · Retail

Black-Friday autoscaling on Kubernetes

The checkout container scales from 3 to 120 pods in ~90 seconds via the Horizontal Pod Autoscaler — something a VM-per-service model (minutes to boot an AMI) can't match. Bin-packing many containers per node keeps cost down off-peak.

Use case · IoT / event processing

Serverless for bursty, fan-in workloads

50,000 devices publish telemetry; a Lambda scales from 0 to thousands of concurrent invocations during spikes and back to zero (you pay only for execution). Long-running stateful services, though, stay on Kubernetes to avoid cold-start and timeout limits.

In the wild

  • Docker/containerd + Kubernetes are the industry default; Helm/Kustomize template manifests and ArgoCD/Flux do GitOps delivery.
  • Monzo and Airbnb run 1,000+ services on Kubernetes; Karpenter autoscales nodes.
  • Serverless: AWS Lambda, Azure Functions, Google Cloud Run, Knative; Coca-Cola and Nordstrom are cited adopters.
  • VM era pioneers: Netflix baked immutable AMIs with Packer and deployed via Spinnaker.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Containers treated as VMs — SSHing in, local stateful storage; breaks restart/scaling semantics.
  • Missing resource limits — noisy-neighbour CPU/memory starvation.
  • Bad probes — liveness that restarts on a slow dependency; readiness that admits traffic too early.
  • Serverless on a latency-critical sync path — cold starts hurt; and costs cross over containers at high sustained RPS.
  • Underestimating K8s — it's powerful but a real operational commitment.

Key takeaways

  • Four options: package, VM, container, serverless — a trade-off of isolation/startup/efficiency/ops.
  • Container + Kubernetes is today's default; serverless shines for event-driven/bursty work.
  • Aim for immutable artifacts + zero-downtime rolling updates.
  • Lean on the platform for discovery, config, probes; a mesh separates release from deploy.
Check yourself

Why can a containerised service scale out in seconds while a service-per-VM model takes minutes — and why does that matter for cost?

Show answer

A container shares the host kernel and starts as a process in a ready-made image (seconds), whereas a VM must boot a full OS (minutes). Faster start means autoscalers can track demand closely, so you run fewer idle instances "just in case." Combined with bin-packing many containers onto each node, that's far higher utilisation — you pay for capacity you actually use rather than over-provisioning slow-to-boot VMs.
