Contract & Component Testing

Trust integrations without slow end-to-end tests — consumer-driven contracts and component tests.

Consumer-driven contract testConsumer-side contract testService component test
In one sentence: Instead of slow, brittle end-to-end tests that spin up the whole system, you test each service in isolation and trust its integrations via consumer-driven contract tests (the consumer writes the expectations, the provider proves it meets them) plus component tests (the service tested alone with its collaborators stubbed).

Use when

  • You have many services and E2E tests are slow/flaky.
  • You want fast feedback that a change won't break consumers.
  • Teams deploy independently and need confidence at the boundary.

Be careful when

  • You rely only on E2E tests (slow, hard to diagnose).
  • Contracts drift from reality because they aren't run in CI.
  • You skip the cheap unit tests at the base of the pyramid.

The problem: end-to-end testing doesn't scale

Verifying behaviour by deploying every service and running tests through the whole stack is slow, expensive, flaky, and — when it fails — hard to diagnose. With dozens of independently deployed services, you need confidence at the boundaries without assembling the entire system every time.

The microservices test pyramid

Push tests down the pyramid: many fast unit tests, fewer integration and contract tests, fewer still component tests (one service in isolation), and a thin layer of end-to-end tests for critical journeys only.

The pattern: consumer-driven contract tests

A contract captures how a consumer expects a provider's API to behave. The consumer writes the expectation (in its own test, producing the contract); the provider replays that contract against the real provider to prove it complies. If the provider ever breaks the contract, the provider's build fails — before deployment.

  • Consumer-side contract test — verifies the consumer's client code against a stub built from the contract.
  • Provider-side verification — verifies the real provider satisfies all its consumers' contracts.
  • Service component test — tests one service end-to-end in isolation, with its dependencies replaced by stubs (often written as Gherkin acceptance scenarios).

Worked use case

Use case · Catching a breaking change in CI

Provider renames a field

Kitchen Service renames ticketId to id. Order Service's published contract still expects ticketId, so Kitchen's contract-verification step fails in CI — the breaking change is caught before deployment, not in a 2 a.m. production incident. No full E2E environment was needed.

In the wild

  • Pact — the leading consumer-driven contract tool: code-first, the consumer test generates a language-neutral JSON contract, shared via a Pact Broker; works across Java, JS, .NET, Go, etc.
  • Spring Cloud Contract — JVM-centric; contracts are written (Groovy/YAML) and used to generate both provider tests and consumer stubs (started provider-driven).
  • Gherkin (Cucumber) — expresses component/acceptance tests in business language.
🧠
Pact vs Spring Cloud Contract

Choose Pact for polyglot, truly consumer-driven workflows (contract generated by the consumer test). Choose Spring Cloud Contract if you're all-JVM and like hand-written contracts that also generate stubs. Both shift integration bugs "left", out of staging and prod.

Pitfalls & anti-patterns

⚠
Watch out for these
  • E2E-only testing — slow, flaky, vague failures; keep E2E thin.
  • Contracts not in CI — a contract nobody runs rots and lies.
  • Over-specified contracts — asserting on incidental fields makes them brittle; assert what you use.
  • Skipping the pyramid base — contract tests don't replace unit tests of your domain logic.

Key takeaways

  • Favour the pyramid: lots of unit + contract + component tests, few E2E.
  • Consumer-driven contracts let providers and consumers deploy independently with confidence.
  • Component tests verify a service alone with stubbed collaborators — fast and deterministic.
  • Tools: Pact (polyglot, consumer-driven), Spring Cloud Contract (JVM).
Check yourself

How does a consumer-driven contract test catch a breaking provider change without deploying both services together?

Show answer

The consumer's expectations are captured as a contract and stored centrally. The provider's CI pipeline replays every consumer contract against the real provider build in isolation. If the provider no longer satisfies a contract, its build fails immediately — so the incompatibility is detected at the boundary, in CI, with no shared end-to-end environment and no waiting for an integration deploy.
