Decompose by Subdomain (DDD)

Use Domain-Driven Design subdomains and bounded contexts to find service boundaries.

Decompose by subdomain
In one sentence: Use Domain-Driven Design to break the problem into subdomains, each with its own bounded context and consistent model — then make each bounded context a service. The boundary between two services is the place where the language of the business changes meaning.

Use when

  • The domain is rich and complex (the same word means different things to different teams).
  • You're already practising (or want) Domain-Driven Design.
  • You need to decide build vs buy per area (core vs generic).

Be careful when

  • You treat a single entity/table as a "bounded context".
  • You try to design the whole model up front instead of discovering it.
  • You let a shared kernel grow until contexts re-couple.

The problem: one "Order" means five different things

In FTGO, the word Order means something different to Order-taking, to the Kitchen, to Delivery and to Accounting. A single, shared "Order" class trying to satisfy everyone becomes a fuzzy, over-coupled mess. DDD gives us a vocabulary for turning these linguistic seams into service boundaries.

The pattern: subdomains & bounded contexts

Domain-Driven Design decomposes the domain (the whole problem space) into subdomains. Each subdomain has a bounded context — a boundary within which a single model and a single ubiquitous language are consistent. One bounded context = one service is the default.

Classify subdomains: core, supporting, generic

Not all subdomains deserve equal investment. DDD classifies them — and that drives build-vs-buy and where your best engineers go:

TypeWhat it isStrategyFTGO example
CoreYour competitive differentiatorInvest heavily, build in-house, best peopleDelivery scheduling / routing
SupportingNecessary but not differentiatingBuild simply, or outsourceRestaurant menu management
GenericA solved commodityBuy / use OSS, wrap with an ACLPayments (Stripe), identity, email

How it works: context mapping

Once you have bounded contexts, you classify the relationships between them (a context map). The relationship dictates the integration style and which team depends on which.

Relationships include Partnership, Customer–Supplier, Conformist, Shared Kernel, Open Host Service / Published Language, and the Anti-Corruption Layer (used whenever you integrate a legacy or generic system without letting its model leak in).

Worked use cases

Use case · Banking

Loan origination — same word, two models

Core: Credit Decisioning (proprietary risk models) with a rich domain model (CreditApplication aggregate, RiskScore value object). Generic: Identity Verification — a thin ACL over a vendor (e.g. Jumio). This classification tells you to staff Credit Decisioning with senior engineers and merely integrate identity.

Use case · Ride-hailing

"Driver" is two different things

In the Matching context a Driver is a lightweight struct (location + availability). In the Driver Management context a Driver is a rich aggregate (profile, documents, earnings, compliance). Same word, two models in two bounded contexts, connected via a Customer–Supplier relationship with an ACL — exactly the seam DDD predicts.

In the wild — tooling & practice

  • Event Storming & Domain Storytelling — workshops to discover subdomains, events and aggregates collaboratively.
  • Context Mapper — an open-source DSL for modelling bounded contexts and their relationships (generates diagrams & service stubs).
  • Microsoft eShopOnContainers — a canonical reference app where each service maps to a DDD bounded context (Catalog, Ordering, Basket, Identity, Payment).
  • Spring Modulith (2023+) — enforces bounded-context boundaries inside a modular monolith and can generate context maps from code; a great stepping stone before extracting services.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Bounded context = one table. A real context encapsulates a business process, not a data structure.
  • Anemic model. Using DDD vocabulary but writing transaction-script logic — overhead without benefit.
  • Shared-kernel sprawl. Too many shared types re-couple the contexts you split.
  • Big design up front. Discover the model iteratively from working code + events, don't pre-design it.

Key takeaways

  • Decompose the domain into subdomains; each bounded context → one service.
  • The service boundary is where the ubiquitous language changes meaning.
  • Classify subdomains core / supporting / generic to direct investment and build-vs-buy.
  • Make context relationships explicit; protect contexts with an anti-corruption layer.
  • This usually agrees with decompose by business capability — use both.
Check yourself

Your "Customer" means different things to Sales (a lead with a pipeline stage) and to Support (an account with tickets). What does DDD say to do?

Show answer

That linguistic divergence is the bounded-context boundary. Don't build one shared Customer model — create two contexts (Sales and Support), each with its own Customer model and language, and integrate them via events or an API (with an ACL if one must conform to the other). Forcing a single model is what creates the fuzzy, over-coupled mess.
