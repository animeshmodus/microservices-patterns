Business Logic: Domain Model & Aggregate

Organise domain logic: transaction script vs domain model, and DDD aggregates as consistency boundaries.

Transaction scriptDomain modelAggregate
In one sentence: Organise a service's domain logic either as simple procedural transaction scripts or as an object-oriented domain model — and in a microservice, structure that model around DDD aggregates, because an aggregate is the unit of consistency you can safely update in one local transaction.

Domain model + aggregates when

  • The business logic is complex and evolving.
  • You need clear consistency boundaries across services.
  • You'll publish domain events or use event sourcing.

Transaction script is fine when

  • Logic is thin CRUD with little behaviour.
  • The service is small and unlikely to grow complex.

The problem: where does the logic live?

Every service has business rules. Two classic ways to organise them:

  • Transaction script — a procedure per request that does the work, with data in dumb structs. Great for simple logic; degrades into tangled spaghetti as complexity grows.
  • Domain model — a web of objects that hold both data and behaviour. Scales with complexity; the foundation of Domain-Driven Design.

But a domain model in a distributed system raises a new question the monolith never had: what set of objects can I update together in a single transaction? Because each service has one database and you can't span a transaction across services, you need explicit consistency boundaries. That's the aggregate.

The pattern: aggregates as consistency boundaries

An aggregate is a cluster of domain objects treated as a single unit. It has a single aggregate root (the only entry point) and a clear boundary. The Order aggregate, for example, owns its OrderLineItems and DeliveryInfo.

The two rules that matter most

  1. Reference other aggregates by identity, not by object reference. An Order holds a consumerId, not a Consumer object. This keeps aggregates loosely coupled and lets each live in its own service.
  2. One transaction updates exactly one aggregate. An aggregate is the consistency boundary. To keep several aggregates consistent (in the same or different services), use a saga driven by domain events — not a bigger transaction.

Aggregate granularity — a design dial

Bigger aggregates give stronger consistency but worse concurrency and bigger transactions; smaller aggregates give better scalability but push more consistency work onto sagas. The book's advice: prefer small aggregates, and reach for sagas to coordinate across them.

Worked use case

Use case · Why "Order" and "Restaurant" are separate aggregates

Designing for independent change & scale

An Order references a restaurantId rather than embedding the Restaurant. That lets Order Service and Restaurant Service be separate services with separate databases, updated in separate transactions. When a restaurant changes its menu, an event flows to Order Service, which updates its own replica — no cross-service transaction needed.

In the wild

  • Aggregates come from Eric Evans' Domain-Driven Design and Vaughn Vernon's "Effective Aggregate Design".
  • jMolecules and Spring Modulith let you express aggregates/entities/value objects in code and verify the rules with ArchUnit tests.
  • The aggregate-per-transaction rule is exactly what makes event sourcing and domain events clean to implement.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Anemic domain model — objects with getters/setters and no behaviour; all logic in "service" classes. DDD vocabulary, transaction-script reality.
  • Object references across aggregates — embedding a Restaurant object inside Order couples them and breaks the one-DB-per-service model.
  • God aggregates — huge boundaries that serialise unrelated updates and create contention.
  • Multi-aggregate transactions — trying to update two aggregates atomically; use a saga.

Key takeaways

  • Simple logic → transaction script; complex logic → domain model.
  • An aggregate is the consistency boundary: one transaction updates one aggregate.
  • Reference other aggregates by id, not by object — that's what lets them live in separate services.
  • Coordinate across aggregates with domain events + sagas.
  • Prefer small aggregates; avoid the anemic-model anti-pattern.
Check yourself

Why must the Order aggregate reference the customer by consumerId rather than holding a Consumer object?

Show answer

Because Consumer is a different aggregate living in a different service/database. Holding a direct object reference would couple the two, imply you could load/update them together, and tempt a multi-aggregate transaction that's impossible across services. Referencing by id keeps the boundary crisp: each aggregate is updated in its own transaction, and cross-aggregate consistency is handled by events/sagas.
