Microservice Security & Access Token

Access token
In one sentence: Authenticate the user once at the edge (the API gateway), then pass a signed, self-contained access token (a JWT) to and between services, so each service can verify identity and authorise the request without a shared session or calling back to a login server.
Use when
  • Requests flow through many services that all need identity.
  • You can't replicate server-side session state everywhere.
  • You want stateless, scalable authorisation.
Be careful when
  • You validate that a token exists but not its issuer/audience/expiry/scope.
  • Tokens are long-lived (hard to revoke).
  • You forget service-to-service auth (mTLS).

The problem: security can't rely on a shared session

In a monolith, an in-memory session handled auth for the whole request. Across services there's no shared session, and making every service call a central auth server on every hop is slow and couples everyone to it. We need identity to travel with the request and be verifiable independently.

The pattern: authenticate at the edge, carry a token

The gateway authenticates the client (via OAuth 2.0 / OpenID Connect) and forwards a signed access token — typically a JWT containing identity and scopes — to downstream services. Each service verifies the token's signature and claims locally and authorises accordingly. No shared session, no callback.

How it works — the modern stack

  • OAuth 2.0 handles delegated authorisation; OpenID Connect (OIDC) adds the identity layer (login/SSO) on top.
  • JWT access tokens are self-contained (claims like sub, scope, exp) and verified by checking the signature against the auth server's public keys (JWKS) — no DB lookup.
  • Short expiry (minutes) limits the blast radius of a leaked token; refresh tokens get new ones.
  • Service-to-service: use mTLS for mutual machine identity (zero-trust), often with SPIFFE/SPIRE issuing short-lived workload certificates — frequently handled by a service mesh.
  • Token exchange (RFC 8693) safely narrows/delegates a token as it moves deeper into the graph.

Worked use case

Use case · Scoped access on a money path

Only "payments:write" may charge a card

A user's JWT carries scope: "orders:read orders:write" but not payments:write. The gateway validates the token; Accounting Service, asked to charge a card, checks the scope and rejects — defence in depth, enforced locally by the service that owns the risk, without a central authorisation call.

In the wild

  • Identity providers: Keycloak (OSS), Auth0, Okta, AWS Cognito, Azure AD/Entra, IdentityServer (.NET).
  • The 2025 production norm: gateway/BFF + OAuth2/OIDC + JWT + mTLS inside the mesh, with SPIFFE/SPIRE for workload identity in zero-trust designs.
  • The BFF pattern is increasingly used to keep tokens out of the browser (server-side session at the BFF, tokens never exposed to JavaScript).

Pitfalls & anti-patterns

⚠
Watch out for these
  • Shallow validation — accepting any well-formed JWT without checking iss, aud, exp and scope. The #1 gateway security gap.
  • Long-lived tokens — hard to revoke; keep them short and use refresh tokens.
  • Trusting the network — no service-to-service auth; an intruder inside the perimeter roams freely. Use mTLS.
  • Secrets in code/images — keep keys/credentials in a secret store (see externalized configuration).
  • Tokens in the browser exposed to XSS — prefer the BFF/server-side-session approach.

Key takeaways

  • Authenticate once at the edge; carry a signed JWT access token through the graph.
  • Use OAuth2/OIDC; verify signature + iss/aud/exp/scope at every service.
  • Keep tokens short-lived; secure service-to-service calls with mTLS (SPIFFE/SPIRE).
  • The gateway/BFF is the natural place to authenticate; a mesh can enforce mTLS for free.
Check yourself

Why is a self-contained JWT preferable to having each service call the auth server to validate a session on every request?

Show answer

A JWT is verified locally by checking its signature against the auth server's published public keys (JWKS) and inspecting its claims — no network call per hop. That keeps services decoupled from the auth server, removes it as a per-request bottleneck/SPOF, and scales statelessly. The trade-off (harder revocation) is managed with short expiry + refresh tokens.

Externalized Configuration

Externalized configuration
In one sentence: Don't bake environment-specific settings (database URLs, credentials, feature flags) into the build — supply them to the service at runtime, either pushed in (env vars / mounted files) or pulled from a configuration/secret server, so one immutable artifact runs unchanged in every environment.
Use when
  • The same build must run in dev / staging / prod.
  • You have secrets that must never live in source or images.
  • You want to change config without rebuilding/redeploying.
Anti-patterns to kill
  • Hard-coded endpoints/credentials in code or the image.
  • Per-environment branches/builds of the same service.
  • Secrets committed to Git.

The problem: config differs per environment, builds shouldn't

A service connects to different databases, brokers and dependencies in dev, staging and prod, and needs secrets it must never expose. If those values are compiled into the artifact, you end up with environment-specific builds, leaked credentials, and "works on my machine" drift. The Twelve-Factor App rule: strictly separate config from code.

The pattern: inject configuration at runtime

Build one immutable artifact; provide its configuration externally when it runs. There are two delivery styles:

  • Push — the deployment platform injects config via environment variables or mounted files when the service starts (e.g. Kubernetes ConfigMaps/Secrets). Simple, no extra runtime dependency.
  • Pull — the service fetches config from a config server or secret store on startup (and can refresh). Centralised, supports dynamic updates and rich secret management.

How it works — and secrets get special treatment

  • Layered config: sensible defaults in the artifact, overridden by environment-specific values.
  • Secrets ≠ config: credentials/keys belong in a secret manager (encrypted at rest, access-controlled, audited, ideally dynamically issued and short-lived) — not in plain ConfigMaps or Git.
  • Refresh: pull-based setups can hot-reload some values (e.g. feature flags) without a restart.
  • This is a core responsibility of the microservice chassis.

Worked use case

Use case · One image, three environments

Promote the exact artifact you tested

The Order Service container built in CI is promoted unchanged from staging to prod. Kubernetes injects the right DB_URL via a ConfigMap and the DB password via a Secret sourced from Vault; the prod database credential is a short-lived, dynamically-issued Vault lease. Nothing environment-specific is ever rebuilt — what you tested is what ships.

In the wild

  • Kubernetes ConfigMaps & Secrets — the default push mechanism for containerised services.
  • HashiCorp Vault — secrets management with dynamic, short-lived credentials and encryption-as-a-service.
  • Spring Cloud Config / Consul KV — centralised pull-based config servers.
  • AWS Parameter Store / Secrets Manager, Azure Key Vault, GCP Secret Manager — managed cloud equivalents.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Secrets in images/Git — the classic breach vector; use a secret manager.
  • Plaintext Kubernetes Secrets — base64 is not encryption; enable encryption-at-rest / external secret stores.
  • Config drift — hand-edited prod config diverging from source of truth; manage config as code.
  • Restart-only changes — no way to flip a flag without redeploy; add refresh where it matters.

Key takeaways

  • Separate config from code — one immutable artifact, runtime-injected settings.
  • Push (env vars / ConfigMaps) is simplest; pull (config server) centralises & enables refresh.
  • Treat secrets specially — a real secret manager (Vault, cloud KMS), never Git or plain images.
  • It's a standard duty of the microservice chassis and ties into security.
Check yourself

Why is promoting the same container image through environments (with externalised config) safer than building a separate image per environment?

Show answer

Because the artifact you tested in staging is byte-for-byte what runs in prod — only the injected configuration differs. Per-environment builds reintroduce the risk that prod runs subtly different code/dependencies than what you validated, and they couple config changes to rebuilds. Externalised config gives you "build once, run anywhere," with environment differences confined to data, not code.

Observability

Health check APILog aggregationDistributed tracingApplication metricsException trackingAudit logging
In one sentence: A distributed system is opaque by default, so make each service observable through six complementary patterns — health checks, aggregated logs, distributed tracing, metrics, exception tracking and audit logging — so you can answer "is it healthy, what's it doing, and why did this request fail?" across dozens of services.
Always — but especially when
  • A single request crosses many services.
  • You need to find which service in a chain failed or slowed.
  • You're running in production (and before you migrate off a monolith).
Don't
  • Bolt observability on after decomposing — add it first.
  • Log without a correlation/trace id — logs become un-joinable.
  • Drown in dashboards no one watches; alert on symptoms users feel.

The problem: where did the request go, and why is it slow?

When "place order" touches gateway → order → kitchen → accounting, a failure or latency spike could be anywhere. You can't SSH into one process and read a stack trace like you would with a monolith. You need the system to emit the signals that let you reconstruct what happened.

The pattern: six ways to see inside

Often summarised as the three pillars — metrics, logs, traces — plus three operational must-haves:

Sub-patternAnswersTypical tech
Health check APIIs this instance alive & ready for traffic?liveness/readiness endpoints; K8s probes
Log aggregationWhat happened? (searchable, centralised)ELK / Grafana Loki
Distributed tracingWhat path did this request take, where was the time?OpenTelemetry + Jaeger / Tempo / Zipkin
Application metricsHow much, how fast, how many errors? (trends)Prometheus + Grafana

How distributed tracing works

A trace follows one request across services. The gateway generates a traceId; each hop adds a span and propagates the id via headers (W3C Trace-Context). Stitched together, the spans show exactly where the time went and which service errored.

🧠
Correlate everything with one id

Put the traceId into every log line. Now a single trace in Jaeger links straight to the exact logs from every service for that request — the difference between a 5-minute and a 5-hour incident.

Worked use case

Use case · The "checkout is slow" page

Find the offending hop

A metrics dashboard shows checkout p99 latency spiking. You open a slow trace: the span breakdown reveals Pricing Service taking 1.8s on a DB call. The trace's logs (same traceId) show a missing index after a deploy. Three pillars, one diagnosis — without guessing.

In the wild

  • OpenTelemetry (OTel) — the CNCF vendor-neutral standard for emitting metrics/logs/traces; in 2025/26 it's the baseline, not optional. The OTel Collector receives, processes and routes telemetry to any backend.
  • Prometheus + Grafana (metrics/dashboards), Jaeger / Grafana Tempo / Zipkin (traces), ELK / Grafana Loki (logs), Sentry (exceptions), Datadog (all-in-one SaaS).
  • Health checks integrate with Kubernetes liveness/readiness probes; a service mesh can emit golden metrics & traces with no app code.

Pitfalls & anti-patterns

⚠
Watch out for these
  • No correlation id — un-joinable logs; you can't follow a request.
  • Liveness probe that calls dependencies — a slow dependency triggers needless restarts; keep liveness local, put dependency checks in readiness.
  • Metric/label explosion — high-cardinality labels blow up Prometheus.
  • Alert fatigue — alert on user-visible symptoms (SLOs), not every twitch.
  • Logging secrets/PII — a compliance incident waiting to happen.

Key takeaways

  • Make services observable via six patterns: health, logs, traces, metrics, exceptions, audit.
  • Distributed tracing + a shared correlation id are what make a request followable.
  • Standardise on OpenTelemetry; route to Prometheus/Grafana/Jaeger/Loki.
  • Add observability before decomposing; a mesh and the chassis give a lot of it for free.
Check yourself

A request fails intermittently somewhere in a 5-service chain. Which observability pattern pinpoints where, and what makes it work?

Show answer

Distributed tracing. A single traceId is generated at the edge and propagated (via headers) to every downstream service, each adding a span. The assembled trace shows the full path and timing, so you can see exactly which hop errored or slowed — something logs/metrics alone can't localise. Putting that traceId in logs lets you jump straight to the relevant log lines.

Microservice Chassis

Microservice chassis
In one sentence: A microservice chassis is a framework (or set of libraries) that handles the cross-cutting concerns every service needs — configuration, health checks, metrics, logging, tracing, security, service discovery — so a new service starts production-ready and developers write business logic, not plumbing.
Use when
  • You're creating many services and don't want to re-solve the basics each time.
  • You want consistency in how services do config/health/metrics.
  • New services should be production-ready on day one.
Be careful when
  • The chassis becomes a shared-library monolith everyone must upgrade in lockstep.
  • It encodes business logic (it should only do plumbing).
  • It conflicts with what a service mesh already provides.

The problem: every service re-implements the same plumbing

Before a service can do anything useful it needs externalised configuration, health checks, metrics, logging and tracing, service discovery and security. Re-writing that for each of 40 services is wasteful and produces 40 subtly different implementations. The first service might take weeks just on plumbing.

The pattern: stand on a chassis

Like a car chassis carries the engine, wheels and electrics so designers focus on the body, a microservice chassis provides the cross-cutting machinery so developers focus on the domain. New services are scaffolded from a template that already wires it in.

How it works — and where it meets the service mesh

Some concerns are best handled in-process by the chassis (config binding, domain-aware health, business metrics). Others — mTLS, retries, traffic routing, basic telemetry — can be pushed out of the process into a service mesh sidecar, keeping them language-agnostic. The modern split:

  • Chassis handles language-specific, application-level concerns (config, domain metrics, structured logging, OTel instrumentation).
  • Service mesh handles network-level concerns (mTLS, retries, circuit-breaking, traffic shifting) for all languages, with no library upgrades.

Worked use case

Use case · A "new service in 10 minutes" template

Golden path scaffolding

A platform team publishes a service template (e.g. a Spring Boot Initializr profile or a Backstage template). Running it yields a service with health endpoints, Prometheus metrics, OTel tracing, config binding and OIDC auth already wired. The developer adds an endpoint and ships — every service in the org looks and operates the same way.

In the wild

  • Spring Boot (+ Spring Cloud) — the archetypal JVM chassis (Actuator health/metrics, config, etc.).
  • Micronaut & Quarkus — fast-startup, low-memory, cloud-native JVM chassis (great for serverless/containers).
  • Go kit / Go-micro — chassis toolkits for Go.
  • Dapr — a sidecar "distributed application runtime" providing service invocation, pub/sub, state and secrets via standard APIs to any language — a chassis you don't link into your code.
  • Backstage software templates standardise how new chassis-based services are created.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Lock-step upgrades — a shared chassis that forces every team to upgrade together recreates monolith coupling; version it and allow gradual adoption.
  • Business logic in the chassis — keep it to plumbing.
  • One chassis per language only — polyglot shops need a strategy (or lean on a mesh/Dapr) so non-JVM services aren't second-class.
  • Reinventing what the platform/mesh gives you — don't duplicate mTLS/retries already in the mesh.

Key takeaways

  • A chassis provides cross-cutting plumbing so teams focus on the domain.
  • It makes new services consistent and production-ready from the first commit.
  • Split responsibilities with a service mesh: app concerns in the chassis, network concerns in the mesh.
  • Tools: Spring Boot, Micronaut, Quarkus, Go kit, Dapr; scaffold via templates/Backstage.
Check yourself

Your org runs services in Java, Go and Python. Why might a service mesh complement (or partly replace) a language-specific chassis here?

Show answer

A chassis is per-language, so each language needs its own implementation of mTLS, retries, circuit-breaking and basic telemetry — duplicated effort and drift. A service mesh handles those network-level concerns in a sidecar outside the process, identically for all three languages and with no library upgrades. The chassis still adds value for in-process, app-level concerns (config binding, domain metrics), so they're complementary.
