Chapter 3 · Interprocess Communication

Synchronous RPI, asynchronous messaging, circuit breakers, service discovery and reliable messaging.

Remote Procedure Invocation (REST & gRPC)

Remote procedure invocation
In one sentence: With Remote Procedure Invocation a service calls another using a synchronous request/response protocol — usually REST over HTTP for public/simple APIs, or gRPC over HTTP/2 for fast internal calls — and waits for the reply.
Use when
  • The caller genuinely needs the reply now to continue.
  • The interaction is a simple query or command with a clear response.
  • You want familiar, debuggable, request/response semantics.
Avoid when
  • You're chaining many synchronous calls (availability collapses).
  • The work is long-running or fire-and-forget — prefer messaging.
  • You need temporal decoupling between sender and receiver.

The problem: services need to talk

The moment you split a monolith, in-process method calls become network calls. The simplest mental model is the remote procedure call: the client sends a request and blocks until the provider responds — just like calling a function, except now there's a network, latency and the possibility of partial failure in between.

The pattern: synchronous request/response

Two technologies dominate:

  • REST — resources + HTTP verbs + JSON. Ubiquitous, browser-friendly, easy to debug, great for public APIs. Describe it with OpenAPI.
  • gRPC — define services/messages in Protocol Buffers, generate strongly-typed clients, transport over HTTP/2 (multiplexed, streaming, binary). Excellent for high-volume internal calls.

The big caveat: synchronous calls reduce availability

This is one of the book's most important lessons. If Order Service synchronously calls Consumer, Kitchen and Accounting to create an order, then Order is only available when all three are up. Availability multiplies downward: chain enough 99.5%-available services and the composite is far worse. Synchronous RPI couples the availability of caller and callee.

⚠
Two defences

(1) Protect every synchronous call with a circuit breaker, timeout and fallback. (2) Where you can, eliminate the synchronous call entirely by using asynchronous messaging — the single biggest availability win in the book.

Worked use case

Use case · Mixed protocols

REST at the edge, gRPC inside

A trading platform exposes REST to partners and browsers (via the API gateway), but services call each other over gRPC for the pricing hot path where every millisecond counts. The gateway translates protocols — the canonical "REST outside, gRPC inside" arrangement.

In the wild

  • Google & Uber run gRPC + Protobuf for vast internal service meshes.
  • Benchmarks consistently show gRPC several times faster than REST for internal traffic due to HTTP/2 + binary encoding.
  • OpenAPI (REST) and Protobuf/Buf (gRPC) anchor API contracts and enable contract testing and codegen.
  • Service location is resolved by service discovery; resilience by circuit breakers (often pushed into a service mesh).

Pitfalls & anti-patterns

⚠
Watch out for these
  • Synchronous call chains — A→B→C→D destroys availability and adds latency. Flatten or go async.
  • No timeouts — a hung dependency exhausts the caller's threads; always bound every call.
  • Chatty fine-grained APIs — many tiny round-trips; design coarser operations.
  • Breaking API changes — evolve compatibly (additive changes, versioning).

Key takeaways

  • RPI = synchronous request/response: REST for public/simple, gRPC for fast internal.
  • Synchronous calls couple availability — minimise chains; protect with timeouts + circuit breakers.
  • Prefer asynchronous messaging whenever a blocking reply isn't truly required.
  • Always describe APIs with a contract (OpenAPI / Protobuf) and evolve them compatibly.
Check yourself

Service A calls B, C and D synchronously, each 99.9% available. Roughly what's A's ceiling availability for that operation, and how would you raise it?

Show answer

≈ 0.999³ ≈ 99.7% — A can't be more available than the product of its hard dependencies. Raise it by removing the synchronous coupling: have A react to events or pre-cache the data it needs, so it can answer even when B/C/D are momentarily down (plus circuit breakers + fallbacks for the calls that remain).

Asynchronous Messaging

Messaging
In one sentence: Services communicate by sending messages through a message broker instead of calling each other directly — the sender doesn't wait for or even know the receiver, which decouples them in time and dramatically improves availability.
Use when
  • You want temporal decoupling — sender and receiver needn't be up together.
  • You're publishing events many services react to.
  • Work is long-running, bursty, or fire-and-forget.
Be careful when
  • The caller truly needs an immediate answer (use RPI).
  • The team isn't ready for eventual consistency & async debugging.
  • You forget idempotency — messages arrive at least once.

The problem: synchronous coupling is fragile

As the RPI lesson showed, synchronous request/response couples the availability of caller and callee and adds latency to every chain. For everything that doesn't need an immediate reply, there's a better way.

The pattern: talk through a broker

The sender writes a message to a channel on a message broker; the broker delivers it to one or more consumers. The sender returns immediately. If a consumer is down, the broker holds the message until it recovers.

How it works

Message & channel types

  • Message kinds: command (do X), document (here's data), event (X happened).
  • Channels: point-to-point (exactly one consumer handles each message — work queues) and publish-subscribe (every subscriber gets a copy — events).
  • Async interaction styles: one-way notifications, async request/response (via a reply channel), and one-to-many publish/subscribe.

Three rules you cannot skip

  • Idempotency. Brokers deliver at least once; design consumers so reprocessing a message is harmless (track processed message ids).
  • Ordering. Use a partition/ordering key (e.g. orderId) so related messages stay in order.
  • Reliable publishing. Don't lose messages on a crash between "DB commit" and "publish" — use the transactional outbox.

Improving availability by removing synchronous calls

The book's flagship example: instead of Order Service synchronously calling Consumer/Kitchen/Accounting at order time, each service subscribes to events and keeps a local replica of the data it needs. Now Order Service can create an order even if the others are momentarily down — availability stops multiplying downward. This is the saga machinery in disguise.

In the wild — brokers & who uses them

  • Apache Kafka — the de-facto log-based streaming backbone (event sourcing, replay, high throughput 500k–1M+ msg/s). As of Kafka 4.0 (2025) ZooKeeper is gone — KRaft makes it a single-binary deploy.
  • RabbitMQ — rich routing (direct/topic/fanout/headers), great for task queues & competing consumers; lowest latency at moderate scale.
  • Apache Pulsar — separates compute (brokers) from storage (BookKeeper); built-in multi-tenancy, geo-replication, tiered storage.
  • NATS / JetStream — tiny, sub-millisecond; CNCF graduated. AWS SQS/SNS, Google Pub/Sub — managed cloud options.
🧠
Choosing a broker

Need ordered, replayable event streams + consumer groups → Kafka. Need flexible routing and classic work queues with no replay → RabbitMQ. Need the lightest possible footprint → NATS. Need multi-region with tight RPOs → Pulsar.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Non-idempotent consumers — duplicate delivery double-charges, double-ships. Track message ids.
  • Lost messages on the dual-write gap — use the outbox.
  • Ignoring ordering — interleaved events corrupt state; key by aggregate id.
  • No dead-letter handling — poison messages block the queue forever.
  • Async everywhere — some flows really do need a synchronous reply; don't be dogmatic.

Key takeaways

  • Messaging decouples services in time → big availability gains over synchronous RPI.
  • Know your message kinds (command/document/event) and channels (point-to-point / pub-sub).
  • Always design for at-least-once delivery: idempotent consumers + ordering keys.
  • Publish reliably with the transactional outbox; it underpins sagas and event sourcing.
Check yourself

A consumer crashes after doing its work but before acknowledging the message. What happens next, and what must the consumer guarantee?

Show answer

The broker, not having received an ack, redelivers the message — so it's processed twice. The consumer must therefore be idempotent: e.g., record each handled message id in the same local transaction as its effect, and skip messages it has already processed. At-least-once delivery makes idempotency non-negotiable.

Circuit Breaker

Circuit breaker
In one sentence: A circuit breaker wraps a remote call and, after too many failures, "trips open" so further calls fail fast (returning an error or fallback) instead of piling up — protecting the caller from a slow or dead dependency and giving that dependency room to recover.
Use when
  • You make synchronous calls to another service or resource.
  • A slow dependency could exhaust your threads and cascade.
  • A degraded/fallback response is better than hanging.
Be careful when
  • You set thresholds blindly — tune to real traffic.
  • You treat it as a substitute for fixing a genuinely broken dependency.
  • Your fallback hides errors that ops needs to see.

The problem: one slow dependency takes everyone down

Synchronous calls have a nasty failure mode. If Kitchen Service becomes slow, every Order Service thread that calls it blocks waiting. Threads pile up, the pool exhausts, and Order Service stops serving all requests — even ones that never needed Kitchen. A single slow dependency triggers a cascading failure.

The pattern: a state machine that fails fast

The circuit breaker tracks the success/failure of calls through it and switches between three states. When failures exceed a threshold it opens and rejects calls immediately (optionally returning a fallback). After a cool-down it goes half-open, letting a trial call through to test recovery.

How it works — the resilience toolkit

A circuit breaker is best combined with its siblings. In Resilience4j the recommended nesting order (outermost first) is Retry → CircuitBreaker → RateLimiter → TimeLimiter → Bulkhead:

  • Timeout — never wait forever; bound every call.
  • Retry — retry transient failures with backoff (but only idempotent operations).
  • Bulkhead — isolate resources (separate thread pools / concurrency limits per dependency) so one drowning dependency can't sink the rest — named after a ship's watertight compartments.
  • Fallback — when open, return a sensible default (cached data, empty list, "try later").

Worked use case

Use case · E-commerce

Recommendations time out — show the page anyway

A product page calls a Recommendations service. When it slows down, the breaker opens and the page instantly falls back to "popular items" from cache. Customers still check out; the recommendation outage is invisible. Without the breaker, every product-page thread would block on Recommendations and the whole storefront would stall.

In the wild

  • Netflix Hystrix pioneered this in the JVM world but has been in maintenance mode since 2018.
  • Resilience4j (v2.x) is its lightweight, functional successor and the default Spring recommends today; it bundles circuit breaker, retry, bulkhead, rate limiter and time limiter.
  • Polly is the equivalent in the .NET ecosystem.
  • A service mesh (Istio/Envoy) can enforce circuit-breaking (outlier detection) at the infrastructure layer, so application code stays clean.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Retrying non-idempotent calls — retried "charge card" = double charge. Combine retries with idempotency.
  • No timeout — a breaker without a timeout still lets slow calls hang.
  • Swallowing failures silently — fallbacks must still emit metrics/alerts; an open breaker is a signal.
  • One global breaker — use a breaker (and bulkhead) per dependency, not one for everything.
  • Thresholds copied from a blog — tune error-rate window and wait-duration to your real traffic.

Key takeaways

  • The breaker turns a slow/dead dependency into a fast, contained failure — no cascading collapse.
  • Three states: Closed → Open → Half-Open, driven by a failure threshold and a cool-down.
  • Always pair it with timeout, retry (idempotent only), bulkhead and fallback.
  • Use Resilience4j (JVM) / Polly (.NET), or push it into a service mesh.
  • Essential for every synchronous call; the gateway is a natural place for it too.
Check yourself

Why does the breaker have a half-open state instead of just flipping straight from Open back to Closed after the timer expires?

Show answer

To probe safely. If it slammed fully Closed, a flood of real traffic would hit a dependency that may still be sick, re-overloading it instantly. Half-Open lets a small number of trial calls through: if they succeed the breaker closes; if they fail it re-opens — so recovery is tested gradually rather than gambled on.

Service Discovery

Self registrationClient-side discoveryServer-side discovery3rd party registration
In one sentence: Service instances come and go (autoscaling, restarts, failures) and their network locations keep changing, so clients can't hard-code addresses — service discovery uses a service registry to track live instances and hand callers a current location.
Use when
  • Service instances are dynamic (cloud, containers, autoscaling).
  • Callers need to find healthy instances at runtime.
  • You want client-side load-balancing or smart routing.
Mostly automatic when
  • You run on Kubernetes — DNS + Services do server-side discovery for you.
  • A service mesh handles it transparently.

The problem: where is the service right now?

In a modern deployment, instance IP:port assignments are dynamic and short-lived. An instance might be on 10.0.3.7:8443 now and gone in five minutes. Hard-coding endpoints or editing config on every scale event is hopeless. We need a live phone-book of "which instances of service X are healthy, and where".

The pattern: a service registry + four micro-patterns

At the centre is the service registry — a database of services, their instances and locations. Around it sit four patterns, split into two questions: how do instances get registered? and how do clients look them up?

Registration: who writes to the registry?

  • Self-registration — each instance registers itself on startup and sends heartbeats. Simple, but couples the service to the registry.
  • 3rd-party registration — a separate registrar watches the deployment environment and registers/deregisters instances. Keeps service code clean (this is essentially what Kubernetes does).

Lookup: who queries the registry?

  • Client-side discovery — the client queries the registry and load-balances itself (e.g. Netflix Eureka + Ribbon). More control, but every client needs discovery logic per language.
  • Server-side discovery — the client just calls a stable router/DNS name; the router does the lookup (e.g. Kubernetes Services, AWS load balancers). Simplest for clients — the modern default.

Worked use case

Use case · Autoscaling

Black-Friday scale-out

Checkout scales from 3 to 120 instances in 90 seconds. With server-side discovery on Kubernetes, each new pod is registered automatically and the checkout Service DNS name immediately load-balances across all 120 — no client changes, no config edits. When pods die, failed health checks deregister them within seconds.

In the wild

  • Kubernetes — built-in service discovery via internal DNS + kube-proxy (server-side, 3rd-party registration). For most teams in 2025 this is their discovery layer.
  • HashiCorp Consul — registry and service mesh; supports both lookup styles, multi-datacenter, rich health checks; integrates with Envoy.
  • Netflix Eureka — the classic client-side registry for Spring Cloud; still used, but its role is shrinking as Kubernetes-native and mesh-based discovery take over.
  • etcd (the store behind Kubernetes), AWS Cloud Map, Nacos — other registries.
🧠
Rule of thumb (2025)

On Kubernetes, use its native discovery (server-side). Outside Kubernetes / multi-platform, reach for Consul. Pick Eureka mainly if you're invested in the Spring Cloud Netflix stack. A service mesh makes discovery invisible to app code.

Pitfalls & anti-patterns

⚠
Watch out for these
  • No health checks / heartbeats — the registry routes traffic to dead instances ("zombie" entries).
  • Treating the registry as always-consistent — registries are eventually consistent; clients must tolerate stale entries (retry + circuit breaker).
  • Reinventing discovery on Kubernetes — running Eureka inside K8s usually just duplicates what the platform already does.
  • The registry as a single point of failure — run it clustered/HA.

Key takeaways

  • A service registry tracks live instances so callers never hard-code locations.
  • Two axes: registration (self vs 3rd-party) and lookup (client-side vs server-side).
  • Server-side + 3rd-party (Kubernetes) is the low-friction modern default.
  • Combine with circuit breakers — discovery data is eventually consistent.
  • A service mesh can make discovery entirely transparent to your code.
Check yourself

Your team runs on Kubernetes but a senior dev wants to add Netflix Eureka "for service discovery". What's your concern?

Show answer

Kubernetes already provides service discovery (DNS-based, server-side, with automatic registration/deregistration via readiness probes). Adding Eureka duplicates that responsibility, adds a component to operate, and forces every service to embed client-side discovery logic. Unless there's a specific cross-cluster/multi-platform need Consul would serve better, lean on the platform's native discovery.

Transactional Messaging (Outbox & CDC)

Transactional outboxPolling publisherTransaction log tailing
In one sentence: To update its database and publish a message reliably, a service writes the message to an outbox table inside the same local transaction, then a separate relay reads the outbox and publishes to the broker — so the two never get out of sync.
Use when
  • A service must change state and emit an event/command atomically.
  • You're implementing sagas or event-driven flows.
  • Losing or duplicating events would corrupt business state.
Be careful when
  • You're tempted to "just publish after commit" — that's the bug this fixes.
  • You forget the consumer side must be idempotent.
  • Ordering across the outbox matters — preserve it.

The problem: the dual-write trap

A service often needs to do two things as one logical step: commit a database change and publish a message. But the database and the broker are two separate systems. If you commit the DB and then crash before publishing, the event is lost forever. If you publish first and then the DB commit fails, you've announced something that never happened. You cannot wrap a normal DB transaction and a Kafka publish in one atomic operation — and 2PC is off the table.

The pattern: Transactional Outbox

Don't publish directly. Instead, in the same local transaction that updates the business tables, also insert the message into an OUTBOX table. Because it's one transaction, the message is saved if and only if the business change is saved. A separate message relay then reads the outbox and publishes to the broker, marking rows as sent.

How it works — two ways to relay

1. Polling Publisher

The relay periodically SELECTs unsent rows from the outbox, publishes them, and marks them sent. Dead simple and portable, but polling adds load and latency, and you must be careful to preserve order and avoid double-sends.

2. Transaction Log Tailing (Change Data Capture)

The relay tails the database's transaction log (e.g. Postgres WAL, MySQL binlog) and publishes each outbox insert as it's committed — no polling, near-real-time, low overhead. This is the modern default, powered by Debezium.

💡
Outbox + Idempotent Consumer = effectively exactly-once

The outbox guarantees at-least-once publishing. Pair it with an idempotent consumer (record each processed message id and skip duplicates) and you get effectively exactly-once business semantics — without distributed transactions.

Worked use case

Use case · Order placement

"Order created" must always reach the kitchen

When Order Service creates an order, it inserts the ORDER row and an OrderCreated outbox row in one transaction. Debezium streams that to Kafka; Kitchen consumes it. Even if Order Service crashes the instant after commit, the event still flows — the kitchen never misses an order, and a redelivery is harmlessly deduplicated.

In the wild

  • Debezium (2.5+ as of 2025) — the industry-standard CDC engine; runs as a Kafka Connect connector, supports Postgres, MySQL, MongoDB, SQL Server, Oracle. It even has a dedicated outbox event router.
  • Kafka Connect — hosts Debezium and routes/transforms the captured events.
  • SeatGeek and many others have publicly described using the transactional outbox to make real-time data distribution reliable.
  • Frameworks like Eventuate Tram provide outbox + relay out of the box (used in the book).

Pitfalls & anti-patterns

⚠
Watch out for these
  • "Publish after commit" code — the exact dual-write bug; one crash and your events lie.
  • Non-idempotent consumers — at-least-once delivery will redeliver; dedupe by message id.
  • Losing order — publish/consume outbox rows in commit order (use ordering keys).
  • Outbox table growth — purge/archive sent rows, or the table balloons.
  • Polling too aggressively — tune interval, or switch to CDC for low latency + low load.

Key takeaways

  • Never dual-write to DB + broker directly — write to an outbox table in the same transaction.
  • A message relay publishes the outbox: polling publisher (simple) or log tailing/CDC (modern, real-time).
  • Debezium is the go-to CDC relay; it has a built-in outbox router.
  • Outbox (at-least-once) + idempotent consumer ⇒ effectively exactly-once.
  • This is the reliability foundation under sagas, domain events and event sourcing.
Check yourself

Why is inserting into an OUTBOX table safe when publishing directly to Kafka after the commit is not?

Show answer

The outbox insert is part of the same local database transaction as the business change, so they commit (or roll back) together atomically — there's no window where one succeeded and the other didn't. Publishing to Kafka is a separate system: a crash between the DB commit and the publish leaves the two permanently inconsistent. The relay then publishes from the durably-committed outbox, so no event is ever lost.
