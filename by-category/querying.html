Querying

Answering queries that span services: API composition and CQRS read models.

API Composition

API composition
In one sentence: To answer a query whose data lives in several services, an API composer calls each owning service and joins the results in memory — the simplest way to do a "cross-service query" when you can't run a database join across separate databases.
Use when
  • A query needs data from 2–4 services.
  • The in-memory join is small and cheap.
  • You want the simplest option before reaching for CQRS.
Avoid when
  • The join is large (big in-memory datasets) or N+1.
  • You need rich filtering/sorting across services → use CQRS.
  • Too many dependencies make the query fragile/slow.

The problem: there are no joins across services

In a monolith, "show order 123 with its restaurant, delivery status and bill" is a single SQL join. Once Order, Kitchen, Delivery and Accounting each own a private database, that join is impossible. The data still needs assembling — so something must gather the pieces.

The pattern: a composer fans out and joins

An API composer implements the query by invoking each provider service and combining the responses. The composer can be a dedicated service, or (very commonly) the API gateway itself.

How it works — design choices

  • Call in parallel wherever the calls are independent (reactive/async), so latency is the slowest single call, not the sum.
  • Handle partial failure — return partial data (e.g. delivery: null) rather than failing the whole response; wrap calls in circuit breakers.
  • Keep the composer stateless — never persist intermediate results (that would make it a hidden shared database).
  • Mind availability — the composer depends on all providers, so its availability is the product of theirs. Cache reference data and degrade gracefully.

Trade-offs

Benefits
  • Simplest way to query across services.
  • No extra datastore to maintain (unlike CQRS).
  • Always reads fresh data from the source.
Drawbacks
  • In-memory joins are costly for large datasets.
  • Reduced availability (depends on every provider).
  • Risk of N+1 fan-out; limited filtering/sorting.

Worked use case

Use case · E-commerce product page

Compose catalog + pricing + inventory + ratings

The composer calls four services in parallel and merges them into one ProductCard. Inventory is treated as optional: on a timeout the page shows the product with stockLevel: null rather than failing — partial results beat a blank page.

In the wild

  • The API gateway commonly performs composition ("gateway aggregation"); Microsoft's Azure Architecture Center documents it as a named pattern.
  • GraphQL (Apollo, Netflix DGS) is essentially typed API composition — the client asks for a shape and the server fans out to resolvers.
  • Netflix's API layer historically fanned out to dozens of services to build a single device payload.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Sequential fan-out — calling providers one after another multiplies latency; parallelise.
  • N+1 composition — composing a list by calling a detail service per item; batch instead.
  • Big in-memory joins — pulling huge result sets to join in the composer; that's a sign you need CQRS.
  • Stateful composer — persisting joined results turns it into a shadow database.

Key takeaways

  • API composition answers cross-service queries by invoking providers and joining in memory.
  • Call providers in parallel, handle partial failure, stay stateless.
  • It's the simplest option — but availability and join cost limit it.
  • When it doesn't scale, switch to CQRS read models; the gateway often hosts the composer.
Check yourself

The "order history" screen must filter and sort thousands of a customer's past orders by fields spread across four services. Why is API composition a poor fit here, and what should you use?

Show answer

Filtering/sorting across services would force the composer to pull large datasets from each provider and join/sort them in memory on every request — slow, fragile, and dependent on all four services being up. This is the textbook trigger for CQRS: maintain a denormalised, query-optimised order-history read model kept current by events, so the query hits one indexed store.

CQRS

Command query responsibility segregation (CQRS)
In one sentence: CQRS splits the model in two — a command side that handles writes through aggregates, and one or more query sides made of purpose-built, denormalised read models (materialised views) that are kept up to date by subscribing to the command side's events.
Use when
  • Queries span services or need a shape the write model can't serve.
  • API composition can't do the filtering/sorting efficiently.
  • You use event sourcing (it can't be queried directly).
  • Read and write workloads need to scale independently.
Avoid when
  • A simple aggregate serves both reads and writes fine.
  • The team can't handle eventual consistency in the UX.

The problem: the write model is a bad query model

Aggregates and event stores are optimised for consistency on write, not for queries. "Show all of a customer's orders, filtered by status, sorted by date, with restaurant names" is awkward against normalised, service-siloed write data — and API composition chokes when the join is large or needs rich filtering. We need a model shaped for reading.

The pattern: separate read models, fed by events

Keep the command side (writes through aggregates). Then build separate read models — each a denormalised view in whatever datastore best fits the query (document, search index, key-value, SQL). A read-model updater subscribes to domain events and keeps the views current. Reads never touch the write store.

How it works — what to get right

  • Idempotent projectors — events arrive at-least-once and can be replayed; track the last processed event id so reprocessing is safe.
  • A view per query — model each read store for its query (Elasticsearch for search, DynamoDB/Redis for key lookups, SQL for reporting).
  • Rebuild by replay — need a new view? Replay the event log to build it; no migration of the write side.
  • Embrace eventual consistency — the read side lags the write side by milliseconds-to-seconds; make that explicit in the API/UX (e.g. 202 Accepted, optimistic UI).
💡
CQRS ≠ Event Sourcing

They pair beautifully but are independent. You can do CQRS with an ordinary relational write DB (publishing events via the outbox / CDC). Event sourcing, however, almost always needs CQRS — because the event store can't answer ad-hoc queries.

Trade-offs

Benefits
  • Efficient queries across services; each view tuned to its access pattern.
  • Read and write sides scale independently.
  • Add new views anytime by replaying events.
  • Natural fit for event sourcing.
Drawbacks
  • More moving parts (projectors, extra stores).
  • Eventual consistency / replication lag.
  • "Read-your-writes" needs explicit handling.
  • Risk of projection sprawl without governance.

Worked use cases

Use case · Order history (the book)

A DynamoDB view of every customer's orders

FTGO maintains an OrderHistory read model in DynamoDB, updated by handlers consuming order/delivery events. The customer's "past orders" screen — filterable, sortable — is a single indexed lookup, no cross-service join.

Use case · Social feed

Pre-built per-user notification feed

A NotificationCreated event fans out into per-user materialised views (a Redis sorted set keyed by user). The read path is a single ZRANGE — the expensive fan-out happens once, on write.

In the wild

  • Axon, Kurrent/EventStoreDB (native projections), Marten — first-class CQRS support.
  • Kafka Streams / ksqlDB and DynamoDB Streams + Lambda — common ways to build projections; read stores are often Elasticsearch, Redis or DynamoDB.
  • Greg Young popularised CQRS; Microsoft's Azure Architecture Center documents it as a core pattern.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Surprise eventual consistency — command then immediate read shows stale data; design the UX for it.
  • Projection explosion — a new view per UI whim; govern and consolidate.
  • Non-idempotent projectors — replays/duplicates corrupt the view.
  • Unmonitored consumer lag — the read model silently falls minutes behind under load.
  • CQRS everywhere — applying it to simple CRUD adds cost with no payoff.

Key takeaways

  • Separate commands (write) from queries (read models); keep views fresh via events.
  • Each read model is denormalised and tuned to one query; rebuild by replay.
  • Make projectors idempotent; embrace and surface eventual consistency.
  • Use it when API composition can't cope, or alongside event sourcing.
Check yourself

A user cancels an order, the command succeeds, but the order list still shows it as "active" for a second. Bug or expected — and how do you handle it?

Show answer

Expected: the read model is updated asynchronously from the cancel event, so it lags briefly (eventual consistency). Handle it in the UX — optimistically update the UI immediately, return 202 Accepted with a way to poll/await freshness, or show a "processing" state — rather than promising read-your-writes from the read side.
