Refactoring to Microservices

Escaping the monolith incrementally with the strangler application and anti-corruption layer.

Strangler Application

Strangler application
In one sentence: Migrate a monolith to microservices incrementally — put a routing facade in front of it and, capability by capability, build new services and divert their traffic to them, so the monolith gradually shrinks until it's "strangled", with no risky big-bang rewrite.
Use when
  • You must modernise a large, running system you can't stop.
  • A big-bang rewrite is too risky/expensive.
  • You want to deliver value & reduce risk continuously.
Be careful when
  • You extract services but keep a shared database (distributed monolith).
  • The migration never finishes — half-extracted zombies pile up.
  • You haven't planned data consistency during the transition.

The problem: you can't stop the world to rewrite

The monolith still runs the business. A "stop everything and rewrite from scratch" project is the classic way to fail — it's high-risk, delivers no value for ages, and the requirements move under you. You need to evolve the system while it keeps serving customers.

The pattern: the strangler fig

Named by Martin Fowler after a vine that grows around a tree until it replaces it. Put a routing facade in front of the monolith. New functionality — and extracted capabilities — become services behind the facade; you route the relevant requests to them. Over time more routes point at services and fewer at the monolith, until the monolith can be retired.

How it works — three complementary strategies

  1. Implement new features as services — stop adding to the monolith; new capabilities go straight into new services behind the facade.
  2. Separate the presentation tier from the backend — split the UI from business logic so each can evolve independently.
  3. Extract business capabilities into services — peel off a capability at a time, including its data, and reroute.

The hard part is the integration glue and data consistency during the overlap: the service and monolith often need each other's data. Techniques include replicating data via events/CDC, dual-writes (carefully), and protecting the new service from the legacy schema with an anti-corruption layer.

🧠
Extract leaf, read-heavy capabilities first

Start with low-coupling, read-heavy slices (e.g. product catalog) — lower risk, quick wins. Save tightly-coupled core domains for when the team has muscle memory. Run new and old in parallel (shadow/dark launch) before flipping traffic.

Worked use case

Use case · E-commerce

Catalog first, checkout last

The facade routes /api/products/* to a new Catalog Service (its own Postgres), seeded via CDC from the monolith. Checkout stays in the monolith for now. After a 3-month parallel run with monitoring, the monolith's catalog tables are retired. The pattern repeats capability by capability until the monolith is gone.

In the wild

  • Martin Fowler coined "Strangler Fig" (2004); it remains the dominant real-world migration strategy.
  • Shopify, Airbnb, Uber all ran multi-year strangler programmes (often with a routing/proxy facade and feature flags).
  • Facade tech: API gateway/Kong/NGINX, AWS ALB path routing, Envoy/Istio traffic splitting; feature flags (LaunchDarkly/Unleash) to flip implementations at runtime.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Shared database left intact — extracting code but not data yields a distributed monolith.
  • Never finishing — zombie half-migrated services accumulate; track and complete each extraction.
  • Dual-write bugs — writing to old and new stores during overlap; use events/CDC and idempotency.
  • God router — routing rules accumulating business logic in the facade.
  • No tests on the monolith before extraction — regressions slip through.

Key takeaways

  • Migrate incrementally behind a routing facade — never big-bang.
  • Three strategies: new features as services, split presentation/backend, extract capabilities.
  • The overlap demands a data-consistency plan (events/CDC) and an anti-corruption layer.
  • Extract low-coupling, read-heavy capabilities first; define "done" so you actually finish.
Check yourself

A team extracts five services from the monolith but they all still read and write the monolith's single shared database. Have they escaped the monolith?

Show answer

No — they've built a distributed monolith. The shared database keeps everything coupled: schema changes ripple across all "services", they can't be deployed truly independently, and they now carry distributed-systems complexity with none of the autonomy benefit. A real strangler extraction moves the data too (each service owns its store), integrating via events/APIs — that's the harder, essential half of the job.

Anti-Corruption Layer

Anti-corruption layer
In one sentence: An anti-corruption layer (ACL) is a translation layer placed at the boundary between a new, clean service and a legacy or external system — it converts between the two models so the messy, foreign model can't leak in and corrupt your domain.
Use when
  • A new service must integrate with a legacy system (e.g. during a strangler migration).
  • You depend on an external/generic system with an awkward model.
  • You want to protect your ubiquitous language at a boundary.
Be careful when
  • The ACL becomes a business-logic dumping ground.
  • You build a bidirectional ACL (often a sign the boundary is wrong).
  • It's a trivial field rename — a value object may suffice.

The problem: the legacy model wants to infect the new one

When a clean new service talks directly to a legacy system, the legacy's data structures, naming and quirks tend to seep into the new code — its DTOs, then its domain objects — until the new service is just a reskin of the old mess. The same happens with external systems that have cryptic or ill-fitting models. You need a guard at the gate.

The pattern: translate at the boundary

From Eric Evans' Domain-Driven Design context-mapping: insert an anti-corruption layer — a set of adapters/translators — between the two bounded contexts. Inside your boundary you speak your clean model; the ACL translates to and from the foreign model. The foreign model never crosses the line.

How it works

  • Put the ACL in the downstream (consuming) service — it's your defence, not the legacy system's job.
  • Translate both structure and semantics — field names, enumerations, units, null conventions, error models — not just shapes.
  • Implement it as adapters/translators (Adapter + Facade); keep your domain model on the inside, pristine.
  • Treat it as plumbing, version it, and test the mappings thoroughly — they're often the most fragile, fiddly code in the system.

Worked use cases

Use case · Healthcare

HL7 → a clean Appointment

A new scheduling service consumes HL7 v2 messages from a legacy EHR. The ACL parses HL7 segments and maps them to a clean Appointment aggregate, so the scheduling domain never imports an HL7 library or knows the cryptic segment names — the legacy format stays outside the wall.

Use case · Strangler migration

New service over a mainframe

During a strangler migration, a new Spring service reads from a CDC-replicated copy of a mainframe table. An ACL maps the mainframe's CUST_REC fields to a clean Customer model, so when the mainframe is finally retired, only the ACL changes — the domain is untouched.

In the wild

  • Defined by Eric Evans (DDD, 2003) as a context-mapping pattern; documented as a cloud pattern by the Azure Architecture Center.
  • Common toolkits: MapStruct (Java), AutoMapper (.NET) for translation; Apache Camel for integration; a Kafka consumer that converts legacy events to clean domain events is an ACL too.
  • Heavily used in SAP S/4HANA migrations, banking mainframe modernisation, and any integration with a generic/vendor subdomain (payments, GDS, identity).

Pitfalls & anti-patterns

⚠
Watch out for these
  • Logic creep — the ACL accumulates business rules; keep it to translation.
  • Bidirectional ACL — translating both ways is a smell that the context boundary is misplaced.
  • Letting types leak — skipping the ACL "just this once" gradually corrupts the clean model.
  • Stale mappings — the legacy schema changes and the ACL silently mistranslates; test & monitor it.
  • Over-engineering — a heavy ACL framework for a couple of field renames.

Key takeaways

  • An ACL is a translation boundary that stops a legacy/external model corrupting your clean one.
  • Place it in the consuming service; translate structure and semantics.
  • Keep it plumbing (adapters/translators), well-tested and versioned — no business logic.
  • Essential during strangler migrations and when integrating generic subdomains.
Check yourself

Why place the anti-corruption layer in the new service rather than modifying the legacy system to emit a clean model?

Show answer

The legacy/external system is often unchangeable, shared by other consumers, or simply not yours to modify — and even if you could, you'd be coupling its release to your needs. The ACL is your protective boundary: by owning the translation in the consuming service, you control your clean model, can adapt when the legacy schema shifts (changing only the ACL), and you don't impose your model on a system that other clients still depend on.
