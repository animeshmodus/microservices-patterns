Deployment

Packaging and running services: containers, Kubernetes, serverless, service mesh & sidecars.

Service Mesh & Sidecar

Service meshSidecar
In one sentence: A sidecar is a helper proxy deployed alongside each service instance that intercepts its network traffic; a fleet of sidecars plus a control plane is a service mesh, which moves cross-cutting network concerns — mTLS, retries, circuit-breaking, routing, telemetry — out of your code and into the infrastructure, for every language at once.
Use when
  • You're polyglot and want uniform networking/security/observability.
  • You want zero-trust mTLS & traffic control without app changes.
  • You run many services on Kubernetes.
Be careful when
  • You have a handful of services — the operational cost outweighs the gain.
  • The team can't operate the control plane (it's a critical component).
  • You'd use the mesh to paper over bad service design.

The problem: the same network code, in every service, in every language

mTLS, retries, timeouts, circuit-breaking, load-balancing, traffic shifting and request telemetry are needed by every service. Implementing them in a per-language chassis means duplicated effort and drift across Java/Go/Python services. What if the platform handled them instead?

The pattern: a proxy sidecar + a control plane

Deploy a sidecar proxy (commonly Envoy) in the same pod as each service. All inbound/outbound traffic flows through it. A central control plane configures every sidecar with policy (security, routing, resilience). Together: a service mesh. Your application code just makes plain calls; the mesh does the rest transparently.

What the mesh gives you (without touching app code)

  • Security: automatic mutual TLS between services (zero-trust), identity, authorization policies.
  • Resilience: retries, timeouts, circuit-breaking (outlier detection), load-balancing.
  • Traffic management: canary & blue-green, weighted routing, fault injection, mirroring.
  • Observability: consistent golden metrics, traces and access logs for every call.
🚀
2024+ : the sidecar-less shift (ambient mesh)

Sidecars cost memory/CPU per pod (an Envoy each). In 2024 Istio Ambient Mesh reached GA, replacing per-pod sidecars with a per-node ztunnel (L4 mTLS) plus optional per-namespace waypoint proxies (L7) — cutting proxy overhead dramatically and removing pod restarts to join the mesh. Cilium pushes mesh features into the Linux kernel via eBPF, aiming for near-zero overhead. The mesh is getting cheaper.

Worked use case

Use case · Zero-trust payments

Lock down who can call the ledger

An Istio AuthorizationPolicy ensures only payment-processor may call ledger, and strict mTLS encrypts all service-to-service traffic — enforced by sidecars, with no change to the services themselves. Security becomes a deployable policy, not scattered code.

In the wild

  • Envoy (born at Lyft) is the de-facto data-plane proxy.
  • Istio — the most featureful mesh (now with Ambient mode); Linkerd — lightweight, Rust-based, simple (Monzo ran it across 1,500+ services); Consul Connect, Cilium (eBPF), AWS App Mesh.
  • Dapr overlaps as a sidecar runtime for app-level concerns (pub/sub, state, invocation).

Pitfalls & anti-patterns

⚠
Watch out for these
  • Sidecar overhead at scale — thousands of Envoys add real RAM/CPU (ambient/eBPF mitigate this).
  • Injection misconfig — a broken sidecar webhook silently drops traffic.
  • Control-plane as SPOF — run istiod HA; it's critical infrastructure.
  • Mesh-as-a-crutch — observability can mask poor API/service design rather than fix it.
  • Adopting too early — a few services don't justify a mesh's operational weight.

Key takeaways

  • A sidecar proxy + control plane = a service mesh that handles networking concerns for you.
  • Get mTLS, resilience, traffic control and telemetry uniformly, across all languages, with no app code.
  • The 2024+ trend is sidecar-less / ambient (Istio Ambient, Cilium eBPF) to slash overhead.
  • Complements the chassis; offloads circuit breaking, discovery, mTLS and tracing.
Check yourself

How can a service mesh give you mutual TLS between all services without any team changing their application code?

Show answer

Because traffic is intercepted by the sidecar proxy, not the app. The control plane issues each workload a short-lived identity/certificate and configures the sidecars to establish mTLS on every connection transparently. The application still makes a plain call to "service-B"; the sidecars terminate/originate the encrypted, mutually-authenticated connection underneath — so encryption and identity are an infrastructure policy, not application logic.

Deployment: Containers, Kubernetes & Serverless

Deploy a service as a containerDeploy a service as a VMLanguage-specific packaging formatServerless deployment
In one sentence: There are four ways to package and run a service — as a language-specific package, in its own virtual machine, in a container (orchestrated by Kubernetes), or as serverless functions — trading isolation, startup speed, resource efficiency and operational burden against each other.
Default choice today
  • Containers + Kubernetes for most long-running services.
  • Serverless for event-driven, spiky, or glue workloads.
  • VM for hard isolation / special hardware; package for legacy/simple cases.
Be careful when
  • You treat containers like VMs (SSH in, local state).
  • You underestimate Kubernetes operational complexity.
  • Serverless cold starts sit on a latency-critical path.

The problem: independently deployable means deployable how?

Microservices promise frequent, independent deployment — but only if packaging and runtime make that cheap, fast and safe. The choice shapes startup time, isolation, density, cost and how much platform you must operate.

How it works — the mainstream path

Most teams build a container image (multi-stage Dockerfile, minimal/distroless base), push it to a registry, and let Kubernetes run it: deployments + replicas, service discovery via DNS, autoscaling (HPA), health probes, and zero-downtime rolling updates. Config & secrets come from ConfigMaps/Secrets. A service mesh can separate release from deploy (ship the bits, shift traffic later).

Worked use cases

Use case · Retail

Black-Friday autoscaling on Kubernetes

The checkout container scales from 3 to 120 pods in ~90 seconds via the Horizontal Pod Autoscaler — something a VM-per-service model (minutes to boot an AMI) can't match. Bin-packing many containers per node keeps cost down off-peak.

Use case · IoT / event processing

Serverless for bursty, fan-in workloads

50,000 devices publish telemetry; a Lambda scales from 0 to thousands of concurrent invocations during spikes and back to zero (you pay only for execution). Long-running stateful services, though, stay on Kubernetes to avoid cold-start and timeout limits.

In the wild

  • Docker/containerd + Kubernetes are the industry default; Helm/Kustomize template manifests and ArgoCD/Flux do GitOps delivery.
  • Monzo and Airbnb run 1,000+ services on Kubernetes; Karpenter autoscales nodes.
  • Serverless: AWS Lambda, Azure Functions, Google Cloud Run, Knative; Coca-Cola and Nordstrom are cited adopters.
  • VM era pioneers: Netflix baked immutable AMIs with Packer and deployed via Spinnaker.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Containers treated as VMs — SSHing in, local stateful storage; breaks restart/scaling semantics.
  • Missing resource limits — noisy-neighbour CPU/memory starvation.
  • Bad probes — liveness that restarts on a slow dependency; readiness that admits traffic too early.
  • Serverless on a latency-critical sync path — cold starts hurt; and costs cross over containers at high sustained RPS.
  • Underestimating K8s — it's powerful but a real operational commitment.

Key takeaways

  • Four options: package, VM, container, serverless — a trade-off of isolation/startup/efficiency/ops.
  • Container + Kubernetes is today's default; serverless shines for event-driven/bursty work.
  • Aim for immutable artifacts + zero-downtime rolling updates.
  • Lean on the platform for discovery, config, probes; a mesh separates release from deploy.
Check yourself

Why can a containerised service scale out in seconds while a service-per-VM model takes minutes — and why does that matter for cost?

Show answer

A container shares the host kernel and starts as a process in a ready-made image (seconds), whereas a VM must boot a full OS (minutes). Faster start means autoscalers can track demand closely, so you run fewer idle instances "just in case." Combined with bin-packing many containers onto each node, that's far higher utilisation — you pay for capacity you actually use rather than over-provisioning slow-to-boot VMs.
