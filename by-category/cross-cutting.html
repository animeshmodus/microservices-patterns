Cross-Cutting Concerns

Concerns every service shares: externalised configuration and the microservice chassis.

Externalized Configuration

Externalized configuration
In one sentence: Don't bake environment-specific settings (database URLs, credentials, feature flags) into the build — supply them to the service at runtime, either pushed in (env vars / mounted files) or pulled from a configuration/secret server, so one immutable artifact runs unchanged in every environment.
Use when
  • The same build must run in dev / staging / prod.
  • You have secrets that must never live in source or images.
  • You want to change config without rebuilding/redeploying.
Anti-patterns to kill
  • Hard-coded endpoints/credentials in code or the image.
  • Per-environment branches/builds of the same service.
  • Secrets committed to Git.

The problem: config differs per environment, builds shouldn't

A service connects to different databases, brokers and dependencies in dev, staging and prod, and needs secrets it must never expose. If those values are compiled into the artifact, you end up with environment-specific builds, leaked credentials, and "works on my machine" drift. The Twelve-Factor App rule: strictly separate config from code.

The pattern: inject configuration at runtime

Build one immutable artifact; provide its configuration externally when it runs. There are two delivery styles:

  • Push — the deployment platform injects config via environment variables or mounted files when the service starts (e.g. Kubernetes ConfigMaps/Secrets). Simple, no extra runtime dependency.
  • Pull — the service fetches config from a config server or secret store on startup (and can refresh). Centralised, supports dynamic updates and rich secret management.

How it works — and secrets get special treatment

  • Layered config: sensible defaults in the artifact, overridden by environment-specific values.
  • Secrets ≠ config: credentials/keys belong in a secret manager (encrypted at rest, access-controlled, audited, ideally dynamically issued and short-lived) — not in plain ConfigMaps or Git.
  • Refresh: pull-based setups can hot-reload some values (e.g. feature flags) without a restart.
  • This is a core responsibility of the microservice chassis.

Worked use case

Use case · One image, three environments

Promote the exact artifact you tested

The Order Service container built in CI is promoted unchanged from staging to prod. Kubernetes injects the right DB_URL via a ConfigMap and the DB password via a Secret sourced from Vault; the prod database credential is a short-lived, dynamically-issued Vault lease. Nothing environment-specific is ever rebuilt — what you tested is what ships.

In the wild

  • Kubernetes ConfigMaps & Secrets — the default push mechanism for containerised services.
  • HashiCorp Vault — secrets management with dynamic, short-lived credentials and encryption-as-a-service.
  • Spring Cloud Config / Consul KV — centralised pull-based config servers.
  • AWS Parameter Store / Secrets Manager, Azure Key Vault, GCP Secret Manager — managed cloud equivalents.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Secrets in images/Git — the classic breach vector; use a secret manager.
  • Plaintext Kubernetes Secrets — base64 is not encryption; enable encryption-at-rest / external secret stores.
  • Config drift — hand-edited prod config diverging from source of truth; manage config as code.
  • Restart-only changes — no way to flip a flag without redeploy; add refresh where it matters.

Key takeaways

  • Separate config from code — one immutable artifact, runtime-injected settings.
  • Push (env vars / ConfigMaps) is simplest; pull (config server) centralises & enables refresh.
  • Treat secrets specially — a real secret manager (Vault, cloud KMS), never Git or plain images.
  • It's a standard duty of the microservice chassis and ties into security.
Check yourself

Why is promoting the same container image through environments (with externalised config) safer than building a separate image per environment?

Show answer

Because the artifact you tested in staging is byte-for-byte what runs in prod — only the injected configuration differs. Per-environment builds reintroduce the risk that prod runs subtly different code/dependencies than what you validated, and they couple config changes to rebuilds. Externalised config gives you "build once, run anywhere," with environment differences confined to data, not code.

Microservice Chassis

Microservice chassis
In one sentence: A microservice chassis is a framework (or set of libraries) that handles the cross-cutting concerns every service needs — configuration, health checks, metrics, logging, tracing, security, service discovery — so a new service starts production-ready and developers write business logic, not plumbing.
Use when
  • You're creating many services and don't want to re-solve the basics each time.
  • You want consistency in how services do config/health/metrics.
  • New services should be production-ready on day one.
Be careful when
  • The chassis becomes a shared-library monolith everyone must upgrade in lockstep.
  • It encodes business logic (it should only do plumbing).
  • It conflicts with what a service mesh already provides.

The problem: every service re-implements the same plumbing

Before a service can do anything useful it needs externalised configuration, health checks, metrics, logging and tracing, service discovery and security. Re-writing that for each of 40 services is wasteful and produces 40 subtly different implementations. The first service might take weeks just on plumbing.

The pattern: stand on a chassis

Like a car chassis carries the engine, wheels and electrics so designers focus on the body, a microservice chassis provides the cross-cutting machinery so developers focus on the domain. New services are scaffolded from a template that already wires it in.

How it works — and where it meets the service mesh

Some concerns are best handled in-process by the chassis (config binding, domain-aware health, business metrics). Others — mTLS, retries, traffic routing, basic telemetry — can be pushed out of the process into a service mesh sidecar, keeping them language-agnostic. The modern split:

  • Chassis handles language-specific, application-level concerns (config, domain metrics, structured logging, OTel instrumentation).
  • Service mesh handles network-level concerns (mTLS, retries, circuit-breaking, traffic shifting) for all languages, with no library upgrades.

Worked use case

Use case · A "new service in 10 minutes" template

Golden path scaffolding

A platform team publishes a service template (e.g. a Spring Boot Initializr profile or a Backstage template). Running it yields a service with health endpoints, Prometheus metrics, OTel tracing, config binding and OIDC auth already wired. The developer adds an endpoint and ships — every service in the org looks and operates the same way.

In the wild

  • Spring Boot (+ Spring Cloud) — the archetypal JVM chassis (Actuator health/metrics, config, etc.).
  • Micronaut & Quarkus — fast-startup, low-memory, cloud-native JVM chassis (great for serverless/containers).
  • Go kit / Go-micro — chassis toolkits for Go.
  • Dapr — a sidecar "distributed application runtime" providing service invocation, pub/sub, state and secrets via standard APIs to any language — a chassis you don't link into your code.
  • Backstage software templates standardise how new chassis-based services are created.

Pitfalls & anti-patterns

⚠
Watch out for these
  • Lock-step upgrades — a shared chassis that forces every team to upgrade together recreates monolith coupling; version it and allow gradual adoption.
  • Business logic in the chassis — keep it to plumbing.
  • One chassis per language only — polyglot shops need a strategy (or lean on a mesh/Dapr) so non-JVM services aren't second-class.
  • Reinventing what the platform/mesh gives you — don't duplicate mTLS/retries already in the mesh.

Key takeaways

  • A chassis provides cross-cutting plumbing so teams focus on the domain.
  • It makes new services consistent and production-ready from the first commit.
  • Split responsibilities with a service mesh: app concerns in the chassis, network concerns in the mesh.
  • Tools: Spring Boot, Micronaut, Quarkus, Go kit, Dapr; scaffold via templates/Backstage.
Check yourself

Your org runs services in Java, Go and Python. Why might a service mesh complement (or partly replace) a language-specific chassis here?

Show answer

A chassis is per-language, so each language needs its own implementation of mTLS, retries, circuit-breaking and basic telemetry — duplicated effort and drift. A service mesh handles those network-level concerns in a sidecar outside the process, identically for all three languages and with no library upgrades. The chassis still adds value for in-process, app-level concerns (config binding, domain metrics), so they're complementary.
